# 11 — Phase 0 Exit Criteria

Build (Phase 1 workflow prototype) may start only when **every** item below is
checked. One page, binary checks, no partial credit.

## Template quality

- ☐ **First prototype claim types selected** (D-02 decided).
- ☐ **Selected-type templates signed off** — for each claim type selected for the
  first prototype: review sheet (doc 04) completed in the workshop by working
  assessors, tested against at least one historical claim file, template v0.2 issued
  with accepted changes, and the named owner has signed off v0.2 (async, within
  1 week of workshop). **If geyser/water and accidental damage are selected (the
  D-02 recommendation), those two sign-offs are mandatory before Phase 1.**
- ☐ **Non-selected templates reviewed as stress-test/reference material** — worked
  through in the workshop where time allows and feedback captured, but they do
  **not block Phase 1**. A non-selected template only becomes a blocking sign-off
  item if and when it is added to the prototype scope.
- ☐ Survey decision taken (D-03) — if surveys are IN, sheets S1/S2 reviewed by a
  surveyor; if deferred, formally parked.

## Workflow validity

- ☐ **Physical-assessment triggers agreed** — trigger matrix (doc 08) confirmed line
  by line in the workshop, T6 rand thresholds filled in, T8 commercial boundary
  defined, assessor-proposed additions incorporated.
- ☐ **Assessor workflow signed off** — the 12-step journey (doc 06) walked through by
  workshop assessors; W7 (checklist-first vs shoot-first capture mode) answered and
  reflected in the spec; no unresolved "that's not how it works" items.
- ☐ **Client journey signed off** — doc 07 reviewed; visibility rule accepted;
  consent placeholder approach accepted for prototype use (with explicit note that
  final wording is production hardening).

## Outputs

- ☐ **Report format approved** — prototype report structure (doc 09) validated:
  two assessors would sign it, one report consumer would accept it, workshop
  questions in 9.1/9.2 answered and applied.
- ☐ **High-res photo flow accepted** — D-05 decided (recommendation: mandatory), and
  assessors confirm the request→capture→confirm-legibility loop matches how they'd
  actually verify plates/serials/documents.

## Decisions & governance

- ☐ **Decision register complete** — all of D-01 … D-10 marked Decided or Deferred
  with owner and date; pilot success definition (D-10) is concrete and measurable.
- ☐ Pilot assessor group named (D-08) and their availability for Phase 1 usability
  sessions confirmed.

## Technical readiness (the only technical work permitted in Phase 0)

- ☐ **Technical spike list confirmed** — and the one mandatory spike executed:
  - **Video provider spike:** shortlist (Daily.co, LiveKit Cloud) tested for:
    iOS Safari + Android Chrome join with no install; rear-camera switching; torch
    control where supported; programmatic frame capture from the remote track;
    client-side full-resolution photo capture + upload. Result: one provider chosen
    for the prototype, with findings written up.
  - Confirmed spike backlog for Phase 1 (not executed yet): PDF generation approach,
    upload-link flow on low bandwidth, capture-to-thumbnail latency on 4G.
- ☐ Reconfirmed in writing: **no production code in Phase 0**; prototype phase uses
  placeholder roles and mock access; Tier B hardening items (doc 02) remain
  documented requirements with a named future gate.
